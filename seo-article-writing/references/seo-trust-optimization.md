# SEO Trust Optimization Skill

Роль: Senior AEO (Answer Engine Optimization) Specialist & Trust/Citation Evaluator.

Навык для аудита и переработки контента в самодостаточные блоки доверия (Extractable Trust Blocks), которые AI-поисковики могут безопасно извлекать и цитировать.

---

## КОНТЕКСТ: КАК AI ИЗВЛЕКАЕТ КОНТЕНТ

AI-поисковики (Perplexity, ChatGPT, Google AI Overviews) не читают страницу последовательно. Они:

1. **Парсят** контент на модульные фрагменты
2. **Ранжируют** фрагменты по extractability и cite-safety
3. **Собирают** лучшие фрагменты в ответ пользователю

AI цитирует не лучший контент, а **наиболее легко извлекаемый и верифицируемый**. Если абзац теряет смысл при вырывании из контекста — AI маркирует его как «рискованный» (low cite-safety) и исключает из генеративной выдачи.

### Статистика

```yaml
cite_stats:
  structured_blocks_citation_rate: "2.8x чаще цитируются"
  stats_heavy_content: "+300% цитирований при 5+ статистиках"
  deep_url_citations: "82% цитирований — глубокие URL"
  seo_overlap: "только 12% AI-цитируемых URL совпадают с топ-10 Google"
  fresh_content_preference: "AI-источники на 25.7% свежее, чем в обычной выдачи"
  zero_click: "69% поисков заканчиваются без клика (2025)"
```

## AUDIT FRAMEWORK: 3 НАПРАВЛЕНИЯ АНАЛИЗА

### 1. Оценка самодостаточности (Extractability Analysis)

Каждый смысловой абзац проверяется по критерию: **сохраняет ли он 100% ясность при чтении отдельно от остального текста?**

#### Что искать

```yaml
blind_spots:                              # «Слепые зоны контекста»
  pronouns_without_context:
    - "он", "она", "оно", "они"           # Кто? Без антецедента — потеря смысла
    - "этот метод", "данный подход"        # Какой именно?
    - "в данном случае"                    # В каком?
    - "it", "this", "these"               # Для EN-контента

  backward_references:
    - "как сказано выше"
    - "как упоминалось ранее"
    - "вышеупомянутый"
    - "см. предыдущий раздел"
    - "продолжая тему..."

  implicit_dependencies:
    # Те же токены правило ai-pattern-cleanup.md (кат. 4) трактует как «пустые
    # связки» — маркер генерации. Список общий, оптика разная: тут риск потери
    # смысла при извлечении AI, там — стилевой тик. Правим по любому обоснованию.
    - Абзац начинается с "Также..." (к чему «также»?)
    - Абзац начинается с "Кроме того..." (кроме чего?)
    - Абзац начинается с "При этом..." (при чём?)
    - Списки без вводного контекста (список чего?)

  incomplete_definitions:
    - Термин используется без определения
    - Аббревиатура без расшифровки
    - Метрика без единицы измерения
```

#### Чек-лист самодостаточности блока

```yaml
extractability_checklist:
  - question: "Блок имеет смысл без окружающего контекста?"
    weight: 2
  - question: "Первые 1-2 предложения дают прямой ответ на вопрос заголовка?"
    weight: 2
  - question: "Нет местоимений без пояснения (он, этот, данный)?"
    weight: 1
  - question: "Нет ссылок на другие части текста (как сказано выше)?"
    weight: 1
  - question: "Все термины определены или общеизвестны?"
    weight: 1
  - question: "Оптимальный объём (40-60 слов для ответа, 120-180 для блока)?"
    weight: 1
  - question: "Содержит конкретику (цифры, даты, имена)?"
    weight: 1
  - question: "Можно использовать как цитату без искажения смысла?"
    weight: 1
  # Максимум: 10 баллов
  # 8-10: высокая extractability
  # 5-7: средняя (требует доработки)
  # 0-4: низкая (переписать)
```

### 2. Аудит безопасности и экспертности (Trust & Safety Assessment)

#### Что искать

```yaml
trust_risks:
  categorical_claims:
    description: "Категоричные утверждения без границ применимости"
    examples:
      - "Kubernetes — лучшая платформа для оркестрации"  # Где доказательство? В каких случаях?
      - "Облако всегда дешевле собственных серверов"      # Не всегда — нужны условия
      - "AI заменит программистов"                        # Слишком категорично
    fix: "Добавить границу применимости и условия"

  ambiguity_risk:
    description: "Фрагменты, которые AI может неверно интерпретировать"
    examples:
      - "Стоимость невелика" (для кого? SMB или Enterprise?)
      - "Быстрое внедрение" (дни? недели? месяцы?)
      - "Высокая производительность" (какие метрики?)
    fix: "Заменить на конкретные цифры или диапазоны"

  reputational_risk:
    description: "Фрагменты, опасные при неполном цитировании"
    examples:
      - "Не используйте X" (без контекста — звучит как запрет)
      - "Технология устарела" (без указания альтернативы и условий)
      - "Безопасность не гарантирована" (без пояснения контекста)
    fix: "Добавить полный контекст в каждый блок"

  unsupported_stats:
    description: "Цифры и статистика без источника"
    examples:
      - "По данным исследований..." (каких?)
      - "Эксперты утверждают..." (какие эксперты?)
      - "Рост составил 40%" (по данным кого? за какой период?)
    fix: "Указать источник, год, методологию"
```

#### Факторы, повышающие cite-safety

```yaml
positive_signals:
  - Конкретные цифры с указанием источника и года
  - Точные формулировки ("средний показатель 15%", не "около 15%")
  - Прозрачная методология (как получены данные)
  - Актуальная дата публикации/обновления
  - Авторская атрибуция с credentials
  - Нейтральный тон без промо-языка
  - Внутренняя непротиворечивость (факты не конфликтуют друг с другом)
```

#### Факторы, снижающие cite-safety

```yaml
negative_signals:
  - Расплывчатые утверждения без доказательств
  - Противоречивые факты внутри одной страницы
  - Промо-контент и opinions-heavy текст
  - Сплошной текст без структуры
  - Устаревшая информация без даты обновления
  - Keyword stuffing (снижает результаты на 10%)
```

### 3. Проектирование Extractable Trust Blocks (Пересборка)

> **Источник фактуры для `evidence`.** Атомарные карточки «fact-cards» (внутренний материал системы, в эту выгрузку не входит)
> — готовая основа поля `evidence`: каждая карточка несёт `claim` + источник + год +
> `confidence` + `allowed_use`. Бери цифру оттуда (или из синтезов `knowledge-base/`),
> а не по памяти. Машинный подбор — `find_fact_cards()` /
> «search_knowledge_base» (внутренняя автопроверка репозитория, в чате недоступна — проверь этот пункт вручную).
>
> Правила подачи:
> - **`allowed_use` обязан включать тип контента** (`seo`, `landing`, `press-release`,
>   `digest`, `speaker-prep`). Вендорские оценки `confidence: low` ограничены `speaker-prep` —
>   в публичный SEO/PR не выносить.
> - **Атрибуция дословно из карточки** (источник + год); цифру не округлять и не подменять.
> - **Просроченную карточку** (`expires` в прошлом, retriever помечает `needs_refresh` и
>   исключает по умолчанию) как факт не использовать. Если такая цифра уже в тексте —
>   `🔢 NUMBER_RECHECK`, повторный прогон фактчекера, а не «освежение» по памяти.
> - **Нет карточки — не выдумывать**: блок переоформляется без новой цифры, недостающее
>   помечается `[NEEDS_FACTCHECK]`.
>
> В pipeline карточки подбирает оркестратор и передаёт `final-trust-editor` во входе
> (чистый контекст — редактор сам не ищет). Контракт — `references/final-trust-editing.md`.

#### Формула Trust Block

Каждый переработанный блок строится строго по формуле:

```yaml
trust_block_formula:
  assertion:                    # [Утверждение]
    description: "Чёткий, однозначный тезис — ответ на вопрос"
    format: "1-2 предложения, без вводных конструкций"
    example: "PostgreSQL поддерживает горизонтальное масштабирование через встроенный механизм партиционирования таблиц начиная с версии 10."

  explanation:                  # [Объяснение]
    description: "Краткое раскрытие — почему это так"
    format: "2-3 предложения с конкретикой"
    example: "Партиционирование разделяет большие таблицы на меньшие физические секции по заданному ключу (диапазон дат, хеш, список значений). Запросы автоматически направляются только к релевантным партициям, что сокращает объём сканируемых данных."

  applicability_boundary:       # [Граница применимости]
    description: "Ограничения — когда работает, когда нет"
    format: "1-2 предложения, начинать с условия"
    example: "Партиционирование эффективно для таблиц от 100 млн строк. Для таблиц менее 10 млн строк накладные расходы на управление партициями могут превысить выигрыш в производительности."

  evidence:                     # [Пример/Источник]
    description: "Фактическое подтверждение — цифра, кейс, ссылка"
    format: "Конкретный факт с атрибуцией"
    example: "По данным Percona (2025), партиционирование таблицы заказов на 500 млн строк сократило время аналитических запросов с 45 до 3 секунд в PostgreSQL 16."
```

#### Типы Trust Blocks

```yaml
block_types:
  answer_capsule:
    description: "Прямой ответ на вопрос заголовка"
    placement: "Сразу после H2/H3 заголовка"
    volume: "30-60 слов"
    purpose: "Для Featured Snippets и AI-цитат"

  fact_block:
    description: "Конкретный факт с атрибуцией"
    structure: "[Факт] + [Источник] + [Год/период]"
    volume: "1-3 предложения"
    purpose: "Для подтверждения утверждений AI"

  comparison_block:
    description: "Структурированное сравнение"
    format: "Таблица или список с конкретными параметрами"
    volume: "5-7 строк таблицы"
    purpose: "Для сравнительных запросов"

  how_to_block:
    description: "Пошаговая инструкция"
    format: "Нумерованный список, каждый шаг самодостаточен"
    volume: "3-7 шагов"
    purpose: "Для инструкционных запросов"

  definition_block:
    description: "Определение термина или концепции"
    format: "**Термин** — определение. [Контекст]. [Граница]."
    volume: "40-60 слов"
    purpose: "Для информационных запросов"

  key_takeaway:
    description: "Ключевой вывод раздела"
    placement: "В конце H2-секции"
    format: "1-2 предложения, резюмирующие раздел"
    purpose: "Для обобщающих AI-ответов"
```

## СТРУКТУРИРОВАННЫЕ ДАННЫЕ (JSON-LD)

Schema.org-разметка — слой disambiguation: она помогает поисковику и AI понять сущности
страницы и делает её eligible для rich results. Это технический артефакт для вёрстки —
помечай как `⚙️ TECH`, не выдавай за редакционную правку.

**Железное правило:** размечай только то, что реально присутствует на странице и совпадает с
видимым контентом. Разметка фактов, которых нет в тексте, или несоответствие — нарушение
structured data policies Google (риск ручной санкции).

Безопасный минимум — `Article`, `FAQPage`, `Organization`. Для авторских страниц — `ProfilePage`,
для навигации — `Breadcrumb`.

```json
{
  "@context": "https://schema.org",
  "@type": "Article",
  "headline": "<H1 страницы>",
  "description": "<краткое описание = intent статьи>",
  "datePublished": "2026-06-07",
  "dateModified": "2026-06-07",
  "author": { "@type": "Person", "name": "<Имя автора>" },
  "publisher": {
    "@type": "Organization",
    "name": "VK Cloud",
    "logo": { "@type": "ImageObject", "url": "https://cloud.vk.com/logo.png" }
  },
  "mainEntityOfPage": { "@type": "WebPage", "@id": "<URL страницы>" }
}
```

```json
{
  "@context": "https://schema.org",
  "@type": "FAQPage",
  "mainEntity": [
    {
      "@type": "Question",
      "name": "<вопрос, дословно как в FAQ на странице>",
      "acceptedAnswer": { "@type": "Answer", "text": "<ответ, дословно как на странице>" }
    }
  ]
}
```

Соответствие: `FAQPage` ↔ FAQ-секция, `Article` ↔ тело статьи, `Organization` ↔ единые
реквизиты бренда (name, url, logo, contactPoint). Проверять разметку — Rich Results Test и
Schema Markup Validator. Rich results не гарантированы даже при валидной разметке.

## CITE-SAFETY SCORING

### Шкала оценки блока (0-10)

```yaml
scoring:
  extractability:       # Вес: 2
    0: "Полностью зависит от контекста"
    1: "Частично самодостаточен, но критичные детали теряются"
    2: "Полностью самодостаточен"

  direct_answer:        # Вес: 2
    0: "Нет прямого ответа, начинается с контекста"
    1: "Ответ есть, но не в первых предложениях"
    2: "Первые 1-2 предложения — прямой ответ"

  factuality:           # Вес: 2
    0: "Общие слова без конкретики"
    1: "Есть конкретика, но без источников"
    2: "Цифры/факты с атрибуцией источника"

  structure:            # Вес: 1
    0: "Сплошной текст"
    1: "Структурирован (списки, таблицы, выделение)"

  volume:               # Вес: 1
    0: "Слишком длинный (>200 слов) или слишком короткий (<20)"
    1: "Оптимальный объём (40-180 слов в зависимости от типа)"

  no_implicit_refs:     # Вес: 1
    0: "Есть имплицитные ссылки (это, выше, данный)"
    1: "Нет имплицитных ссылок"

  neutrality:           # Вес: 1
    0: "Содержит промо-язык или необоснованные оценки"
    1: "Нейтральный, фактологический тон"
```

### Интерпретация общего Score

```yaml
interpretation:
  8-10: "Высокая cite-safety. Блок готов к цитированию AI-системами."
  5-7: "Средняя cite-safety. Рекомендуется доработка перед публикацией."
  0-4: "Низкая cite-safety. Блок необходимо полностью переработать."
```

## ПРИМЕР АУДИТА

### Исходный текст (cite-safety: 3/10)

> Как мы уже обсуждали, это решение значительно ускоряет работу. По мнению экспертов, данный подход является оптимальным для большинства компаний. При этом стоимость невелика.

### Выявленные проблемы

1. «Как мы уже обсуждали» — обратная ссылка, теряется при извлечении
2. «это решение» — нет указания, какое именно
3. «значительно ускоряет» — насколько? нет цифр
4. «По мнению экспертов» — каких экспертов?
5. «данный подход» — какой подход?
6. «является оптимальным» — категоричное утверждение без границ
7. «для большинства компаний» — каких? размер, отрасль?
8. «стоимость невелика» — сколько?

### Оптимизированный Trust Block (cite-safety: 9/10)

> **Managed Kubernetes сокращает время развёртывания кластера с 2-3 дней до 15 минут.** Провайдер берёт на себя обновления control plane, мониторинг и резервное копирование etcd. По данным CNCF Survey (2025), 78% компаний с командой DevOps до 5 человек выбирают managed-решения. Подход оправдан для команд, которым критично время выхода на рынок. Для компаний с собственной платформенной командой (10+ инженеров) самостоятельное управление кластером может дать больше гибкости.

## ПРОФИЛЬ: ДЗЕН (лента + AI-поисковики)

> Парный слой: голос и жанры канала — «vk-tech-dzen-voice» (внутренний материал системы, в эту выгрузку не входит)
> (режимы «канал/спикер», ЛОНГ/ФИЧ/ВИДЕО/ЭКСПЕРТ). Здесь — только
> cite-safety/извлекаемость; для Дзен-материала применяются оба слоя.

На Дзене работают те же Trust Blocks, но с поправкой на формат ленты: текст компактный (3000–5000 знаков), а дочитывания важнее, чем плотность цитируемых блоков. Поэтому **ETB-формат обязателен не для каждого абзаца, а только для ключевых смысловых блоков** — переходы, открытые петли и CTA остаются «живыми» и в ETB не переписываются (иначе текст теряет разговорный тон и роняет дочитывания).

### Где ETB обязателен

| Элемент статьи | ETB | Почему |
|----------------|-----|--------|
| Лид (первые 2–3 предложения) | **Да** | чаще всего цитируется как «суть статьи» |
| Определения и ключевые тезисы | **Да** | извлекаются AI для прямых ответов |
| Рекомендации (actionable advice) | **Да** | цитируются как готовый совет |
| Сравнения и выводы | **Да** | уходят в таблицы и сводки AI |
| Переходные абзацы, открытые петли | Нет | связки между секциями, не цитируются |
| CTA и призыв к подписке | Нет | нерелевантны для AI-выдачи |

### Дзен-специфика cite-safety

- **Источник внутри блока.** Дзен понижает за устаревшие данные, а AI-поисковики предпочитают свежее: `«В 2024 году рынок вырос»` → `«По данным CNews Analytics (2025), рынок вырос на 25%»`.
- **Именованные сущности вместо «решения/продукта».** В ленте абзац вырывается без заголовка: «этот мессенджер» теряет смысл, «Мессенджер VK WorkSpace» — нет.
- **Двойной адресат.** Статья индексируется в Яндексе и попадает в Яндекс Нейро и Перплексити: первый экран даёт ответ, а не подводку (перевёрнутая пирамида).

### Пример ETB для Дзена

**Было (low cite-safety):**
> Этот мессенджер гораздо безопаснее. Как мы говорили выше, есть несколько причин для перехода. Он поддерживает нужные интеграции и вообще лучше подходит для работы.

**Стало (ETB):**
> **Корпоративный мессенджер безопаснее публичного для рабочей переписки.** Корпоративные решения поддерживают DLP-интеграцию, журналирование сообщений и контроль утечек — функции, недоступные в публичных мессенджерах. Это актуально для компаний от 50 сотрудников, работающих с персональными данными (152-ФЗ). Например, Мессенджер VK WorkSpace интегрируется с InfoWatch, Solar Dozor и SearchInform.

Подключается через platform-скилл `dzen` (модуль media-writer) — он ссылается на этот профиль в чеклисте перед публикацией.

## КОНТРАКТ С ГОЛОСОМ КАНАЛА (ETB vs живой голос)

Две цели тянут текст в разные стороны: cite-safety — к атомарным
самодостаточным блокам, голос канала (`voice-packs`, верхняя граница тона) —
к характерной живой интонации. До сих пор конфликт был примирён только для
Дзена (профиль выше); ниже — явный контракт для остальных каналов. Общий
принцип: **ETB-дисциплине подчиняются фактурные блоки, голосу — связки,
переходы, лид и CTA.** Извлекаемость не означает «весь текст звучит как
энциклопедия».

| Канал | Что переводим в ETB | Что остаётся живым (голос ведёт) |
|-------|---------------------|----------------------------------|
| SEO-статья / техстатья | определения, цифры с атрибуцией, ответы FAQ, первые 1–2 предложения секций | лид-сцена, переходы, примеры-истории |
| Колонка (C-level) | 2–4 справочные врезки: цифра + источник, определение термина | весь остальной текст — позиция, опыт, спор; колонку НЕ атомизируем: мнение не обязано быть извлекаемым |
| Лендинг | блоки фактов: SLA, сертификации, лимиты, цены — самодостаточно и с атрибуцией | боль→решение, выгоды, CTA — по паспорту канала №4 |
| Пресс-релиз | лид с цифрой и заключительный факт-абзац (уже ETB по фирменной структуре) | цитата спикера — неприкосновенна, её не переписываем «для извлекаемости» |
| Телеграм | одна проверяемая цифра с источником на пост — максимум | всё остальное; телеграм — не ETB-канал |
| Анонс мероприятия | блок «что/когда/где/для кого» — самодостаточный | приглашение, мотивация прийти |

Правило приоритета при неразрешимом конфликте в конкретном абзаце: в
фактурном блоке выигрывает cite-safety, в голосовом — голос; спорный абзац
помечается на ручной просмотр, а не «усредняется» до безликого. Red flags
дрейфа между каналами — «channel-style-drift» (внутренний материал системы, в эту выгрузку не входит).

## ИНТЕГРАЦИЯ С АГЕНТАМИ

Навык подключается к нескольким агентам системы. При активации в рамках любого агента:

1. Разбить сгенерированный контент на смысловые блоки (по H2/H3)
2. Оценить каждый блок по Cite-Safety Score
3. Показать таблицу аудита с проблемными местами
4. Предложить переработанные Trust Blocks
5. Спросить, какие блоки заменить
6. Пересобрать контент с обновлёнными блоками
7. Показать итоговый Cite-Safety Score

integrated_agents:

### Рекомендации при генерации

Если навык активен, при генерации контента дополнительно:

```yaml
generation_enhancements:
  - Начинать каждый H2-раздел с Answer Capsule (30-60 слов)
  - Не использовать имплицитные ссылки между разделами
  - Каждый абзац — максимум 3 предложения
  - Статистики с указанием источника и года
  - Завершать каждую H2-секцию Key Takeaway
  - Таблицы для сравнений вместо текстовых перечислений
  - Определения терминов в формате: **Термин** — определение
```
