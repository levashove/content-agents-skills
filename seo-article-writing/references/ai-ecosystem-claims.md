# Источники и атрибуция в темах AI-экосистемы

Доменное правило для тем про AI-инфраструктуру и экосистему: LLM и базовые
модели, ИИ-агенты и мультиагентные системы, фреймворки (LangChain / LangGraph /
AutoGen / CrewAI и др.), протоколы (MCP, function calling / tool use), inference
(vLLM / Ollama / TGI), RAG, версии и статусы релизов. Подключается автоматически
по триггерам темы (`DOMAIN_RULE_TRIGGERS`) — зеркало `cybersecurity-claims` для
второй ниши с концентрацией галлюцинаций.

## Зачем правило

AI-экосистема меняется быстрее любого другого домена системы: версии мажорятся
за месяцы, проекты переименовываются, «стандарты» рождаются и умирают за
полгода. Память модели здесь устаревает ещё до релиза самой модели, а вторичных
пересказов больше, чем первоисточников. Зафиксированный кейс (отчёт
«2026-07-12-enterprise-ai-agent-architecture-report» (внутренний материал системы, в эту выгрузку не входит)):
автор уверенно написал «Microsoft AutoGen достиг версии 1.0 GA к 2026» —
несуществующий факт, снятый фактчекером. Класс дефекта тот же, что был в ИБ до
`cybersecurity-claims` (5 из 6 галлюцинаций мая — security-темы): убедительная
конкретика без первоисточника.

## Главный принцип: версия/статус — только из release notes

Любое утверждение вида «X достиг версии N», «фича Y в GA/beta/deprecated»,
«проект Z переименован/архивирован/поглощён» — **проверяемое релизное
утверждение**, а не фоновое знание. Ему нужен первоисточник — официальный
changelog / release notes / анонс проекта — с датой. Нет пруфа →
`[NEEDS_FACTCHECK: версия/статус не подтверждены release notes]`, а не
уверенная формулировка «по памяти».

## Белый список классов источников (Tier 1 по шкале фактчекера)

| Класс | Примеры | Для чего |
|-------|---------|----------|
| Официальные release notes / changelog / GitHub Releases проекта | releases-страницы фреймворков и моделей | версии, GA/beta/deprecated, breaking changes |
| Официальные блоги и доки вендоров моделей | блоги/доки OpenAI, Anthropic, Google (DeepMind/AI), Meta AI, Mistral, Сбер (GigaChat), Яндекс (YandexGPT) | возможности моделей, лимиты, цены, анонсы |
| Спецификации протоколов и их репозитории | спецификация MCP, OpenAPI-описания function calling | что стандарт реально определяет |
| Модельные карточки и лидерборды с методологией | Hugging Face model cards, публичные бенчмарки с описанием сетапа | сравнение моделей — только с указанием бенчмарка и даты |
| Рецензируемые/препринт-работы | arXiv (с оговоркой «препринт»), конференции (NeurIPS, ICML, ACL) | научные утверждения, методы |
| Отчёты аналитиков об adoption | Gartner, IDC, Stanford AI Index, опросы CNCF/StackOverflow | рыночные цифры — с годом и периметром выборки |

## Запрещено без перепроверки первоисточника

- **Агрегаторы и «новости ИИ»** (дайджест-сайты, Telegram-пересказы, SEO-блоги
  «top-10 LLM frameworks») — как источник версий, статусов и цифр. Только как
  наводка на первоисточник.
- **Вендор-блог о чужом опросе** — год берётся у самого опроса, не у блога
  (рецидив «CNCF FinOps 2026» вместо 2024 — `source-attribution.md`).
- **Бенчмарки без методологии** («модель X на 40% лучше Y») — без названия
  бенчмарка, даты и сетапа цифра не используется.
- **Экстраполяция роадмапа** («к концу года выйдет…», «планируется GA») как
  свершившийся факт — только с пометкой «по роадмапу проекта на {дата}».

## Специфика свежести (поверх `freshness-check.md`)

Для AI-экосистемы действуют самые короткие окна из таблицы freshness-check:

- версии фреймворков/моделей и статусы фич — **на момент публикации**
  (обязательная сверка с release notes при фактчеке);
- GitHub stars / downloads — не старше 3 месяцев, с датой замера;
- цены и лимиты API — на момент публикации, со ссылкой на прайс-страницу;
- «проект мёртв/жив» — сверять по активности репозитория и анонсам, не по
  памяти (проекты архивируются и возрождаются за квартал).

## Контракт для автора

1. Пишешь версию, статус (GA/beta), дату релиза, цену, лимит — рядом дата и
   источник из белого списка классов, иначе `[NEEDS_FACTCHECK]`.
2. Название проекта/протокола — в каноническом написании из его репозитория
   (не «МСП» вместо MCP, не «Autogen» вместо AutoGen).
3. Сравнение моделей/фреймворков — только по именованному бенчмарку с датой
   или явно как субъективный опыт («в нашем сценарии…»).
4. Прогнозы и роадмапы — с пометкой оценочности и датой снимка.

## Контракт для фактчекера

1. Каждое релизное утверждение (версия/статус/дата) — сверить с release notes
   проекта; несуществующая версия снимается, не смягчается.
2. Цифры adoption/рынка — развернуть до первоисточника (опрос/отчёт), сверить
   год и периметр выборки (`source-standards.md`, «Семантика метрик»).
3. Утверждения о протоколах («MCP умеет…») — сверить со спецификацией, а не с
   пересказом в блоге.
4. Источник вне белого списка классов — `[NEEDS_FACTCHECK: источник за
   пределами белого списка AI-экосистемы]`.
