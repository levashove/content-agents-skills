# Пресс-релиз VK Tech (чат-версия конвейера)

Композитный скилл: фирменная структура, эталонные релизы и редполитика
пресс-релизов VK Tech в одном процессе для работы в чате. Этот файл —
маршрутизатор: жёсткий минимум и порядок шагов здесь, детали — в Knowledge
(открывай по карте ниже, а не все сразу).

**Границы применения.** Скилл — для написания пресс-релиза по фактуре
пользователя. В чате нет изолированного ресёрчера и фактчекера конвейера — их
заменяет самопроверка шага 7; это честное ограничение, а не эквивалент полного
конвейера.

## Жёсткий минимум (действует всегда, даже без открытия справочников)

1. **Факты — только из фактуры пользователя.** Продуктовые цифры, сертификации,
   сроки и условия не выдумываются и не «вспоминаются»: нет в предоставленных
   материалах — спроси или поставь маркер `[NEEDS_FACTCHECK: что проверить]`.
   Расхождение фактуры с известными фактами — флаг, а не молчаливый выбор.
2. **Фирменная структура 2026:** дата → заголовок → **лид курсивом** → тело →
   **цитата спикера** → [вторая цитата] → **заключительный факт-абзац** →
   [бойлерплейт] → [«Почитать по теме»]. Три элемента жирным — обязательные.
3. **Лид — курсивом, 1–2 предложения, с главной цифрой релиза.** Партнёрский
   релиз/меморандум может быть без цифры — суть в самом факте договорённости.
   Повод/площадка анонса — в лиде, если новость привязана к событию.
4. **Цитата открывается сильнейшим релевантным тезисом** (рыночная цифра — один
   из вариантов, не обязательный), затем проблема → решение → ценность.
   60–150 слов. Должность спикера — канон из фактуры, не из памяти.
5. **Без буллитов в теле релиза** — сплошной текст. Сложные термины
   расшифровываются при первом упоминании: релиз читают журналисты.
6. **Сравнительная метрика — всегда с базой сравнения.** «Ускоряет на 15%» без
   «по сравнению с {база}» — искажение. Сертификации — точно и дословно.
7. **Один заголовок в выдаче.** Варианты заголовка/лида живут на шагах выбора,
   в тело релиза не попадают. Без запрещённых слов и ИИ-паттернов.

## Процесс

### Шаг 1. Вход

Собери от пользователя: инфоповод, тип релиза (продуктовый / партнёрский /
мероприятие / аналитика), фактуру (цифры, сертификации, сроки, условия),
спикера с точной должностью, повод/площадку анонса. Чего-то нет — спроси одним
сообщением списком, не по одному.

### Шаг 2. Заголовок

Предложи 3–5 вариантов по формуле `[Компания] + [глагол] + [продукт] + [для
кого/зачем]`, ≤ 100 символов, без превосходных степеней — и дай выбрать.
Формулы и эталонные заголовки → `press-release-knowledge.md`.

### Шаг 3. Лид

К выбранному заголовку — 2–3 варианта курсивного лида с главной цифрой
(и поводом анонса, если есть) → на выбор. Требования к лиду →
`press-release-standards.md`.

### Шаг 4. Тело

Механика сплошным текстом: функционал → как работает (термины расшифровать) →
аудитория/сценарии → варианты поставки. Голос делового релиза и запреты →
`press-release-content-quality.md`.

### Шаг 5. Цитата спикера

Открытие сильнейшим тезисом, формат: *«{текст}»*, — комментирует **{Имя
Фамилия}, {должность}**. Формулы и эталонные цитаты →
`press-release-knowledge.md`.

### Шаг 6. Финал

Заключительный факт-абзац (сертификации, условия, сроки, доступность) +
бойлерплейт по типу релиза (каталог бойлерплейтов →
`press-release-knowledge.md`).

### Шаг 7. Самопроверка

Пройди чек-лист из `press-release-standards.md`. Стоп-слова и
замены → `press-release-content-quality.md` и
`content-quality.md`; продуктовые названия →
`press-release-brand-terminology.md`, `brand-terminology.md`
и `vk-tech-redstandards.md` (типографика); атрибуция цитат и чужих
исследований → `source-attribution.md`; чистка ИИ-паттернов →
`ai-pattern-cleanup.md`.

### Шаг 8. Сдача

Отдай релиз + отдельным блоком: выбранные заголовок и лид, оставшиеся маркеры
`[NEEDS_FACTCHECK]` (если есть) и что человеку проверить руками (должность
спикера, продуктовые цифры, сертификации). Релиз с маркерами не выдавай за
готовый к публикации.

в чате их заменяет шаг 7.
