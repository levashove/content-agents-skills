# Стандарты качества контента

Эти правила действуют ВСЕГДА для любого контента. Полный справочник → навык `vk-tech-redstandards`.

## Инфостиль

- Каждое утверждение подкреплено фактом, цифрой или примером
- Нет воды: если предложение можно удалить без потери смысла — удали
- Нет канцеляризмов и штампов (полный список → навык `vk-tech-redstandards`)
- Тон: практик-эксперт, не пресс-служба

## Первый абзац

- Начинай с факта, провокации, цифры или задачи читателя
- НИКОГДА не начинай с «В современном мире...», «На сегодняшний день...», «В эпоху цифровизации...»
- Первое предложение — крючок, который удержит читателя

## Запрещённые слова

Никогда не используй: инновационный, революционный, уникальный, является, осуществлять, позволяет, даёт возможность, на сегодняшний день, данный, следует отметить, очень, весьма, абсолютно, действительно, мы рады сообщить, передовой. Запрет — на лексему во всех словоформах («позволило», «позволяют» — тоже), не только на точную форму.

Без подтверждения цифрами: ведущий, лидирующий, лучший, идеальный, оптимальный.

### Стоп-лист и голос спикера/канала (приоритет)

Стоп-лист правит всю прозу автора, **включая новые (сгенерированные) цитаты спикеров**: голос спикера — это тезисы, синтаксис и фирменные рамки профиля, а не запрещённая лексика («позволяет» из опорного тезиса → конкретный глагол). Исключения: (1) реальные опубликованные цитаты неприкосновенны (`final-trust-editing.md`); (2) фирменная формула, явно зафиксированная в профиле спикера как обязательная рамка, допустима внутри его цитаты даже против продуктового стоп-листа — вне цитат исключений нет; (3) упоминание-отрицание клише («не …„под ключ"») — допустимо, но не приём по умолчанию.

## Замены клише

| Клише | Замена |
|-------|--------|
| осуществлять деятельность | работать |
| производить оплату | платить |
| является лидером | занимает N% рынка |
| уникальное решение | решение с [конкретная характеристика] |
| комплексный подход | [перечисли что входит] |
| значительно выросла | выросла на X% |
| существенно ускоряет | сокращает время с X до Y |
| даёт возможность | конкретный глагол |
| позволяет | конкретный глагол |

## Факты

- Не придумывай статистику, даты, цитаты
- Если данных нет — напиши «[нужны данные: описание]», не выдумывай
- Каждый факт подкрепляй данными, примером или ссылкой

## Обязательные этапы pipeline

Фактчек — не опциональный шаг. Там, где он проводился, находилось от 1 до 5 исправлений на материал (данные по отчётам 2026-Q1).

| Тип контента | Ресёрч | Фактчек | Редактура (опц.) | Trust-аудит (опц.) | Финальная траст-редактура |
|--------------|--------|---------|------------------|--------------------|---------------------------|
| SEO-статья | обязательно | **обязательно** | по запросу | поглощена финалом | **обязательно** |
| Пресс-релиз | обязательно | **обязательно** | по запросу | поглощена финалом | **обязательно** |
| Колонка / экспертный материал | обязательно | **обязательно** | по запросу | поглощена финалом | **обязательно** |
| VK Cloud статья | обязательно | **обязательно** | по запросу | поглощена финалом | **обязательно** |
| Дайджест | обязательно | опционально (сборка из первоисточников; Tier-шкала — digest-standards.md) | по запросу | поглощена финалом | **обязательно** |
| Email-рассылка | по запросу | по цифрам | по запросу | — | **обязательно** |
| Telegram-пост | по запросу | по цифрам | по запросу | — | **обязательно** |
| Лендинг | по запросу | по цифрам | по запросу | — | **обязательно** |
| Страница мероприятия | по запросу | по цифрам | по запросу | — | **обязательно** |
| Интервью (CNews и др.) | обязательно | **обязательно** | по запросу | поглощена финалом | **обязательно** |

**Финальная траст-редактура** — отдельный субагент `final-trust-editor` (shared-subagents). Запускается **после** фактчекера и **перед** сохранением. Объединяет trust-оптимизацию и полировку стиля. Жёстко защищает числа, ссылки, сноски и атрибуции от изменений. Полный контракт — «final-trust-editing» (внутренний материал системы, в эту выгрузку не входит).

Если пользователь явно отказывается от фактчека в обязательном этапе — оркестратор фиксирует это в отчёте как `Фактчек: пропущен по запросу пользователя` и предупреждает о риске непроверенных фактов.

### Автозапуск trust-аудита

Отдельный шаг автозапуска `trust-optimizer` упразднён. Trust-оптимизация теперь часть обязательного финального шага `final-trust-editor`, который запускается всегда после фактчекера. Если фактчекер выставил Cite-Safety Score < 8 — финальный редактор работает с повышенным фокусом на переработку Trust Blocks, при Score ≥ 8 — ограничивается полировкой стиля.

## Приоритет актуальности данных

> **Канон порогов свежести — «freshness-check» (внутренний материал системы, в эту выгрузку не входит)** (лимиты по
> типам данных, чек-лист фактчекера). Этот раздел — рабочая выжимка для
> автора/ресёрчера; при расхождении порогов выигрывает freshness-check,
> правки порогов вносятся сначала туда.

При сборе информации отдавай приоритет свежим источникам:

1. **Последние 6 месяцев** — основной приоритет
2. **Последний год** — если за 6 месяцев данных недостаточно
3. **1–3 года** — для контекста и трендов
4. **Старше 3 лет** — только для исторической справки, с явной пометкой о давности

Если используешь данные старше года — укажи дату явно: «по данным за 2024 год», «в исследовании 2023 года».

### Приоритет для числовых данных

Для любой цифры (объём рынка, доли, MAU/DAU, GitHub stars, темпы роста, прогнозы, CAGR) действует жёсткая приоритизация по году публикации первоисточника:

1. **Текущий год** — основной приоритет. Если первоисточник за текущий год существует, используй его.
2. **Прошлый год** — допустимо, если за текущий год данных ещё нет или выборка неполная. Обязательно указывай год в тексте: «по данным за {год-1}».
3. **Позапрошлый год** — только как контекст динамики («с X в {год-2} до Y в {год-1}»), не как актуальное значение.
4. **Старше 2 лет** — запрещено для цифр, описывающих текущее состояние рынка, продукта, пользовательской базы. Допустимо только для исторических сравнений с явной датой.

Исключения: цифры, которые физически не обновляются ежегодно (разовые события, капитализация раунда, дата релиза). Для таких — указывать точную дату события.

Если ресёрчер находит данные только за позапрошлый год — это красный флаг: либо метрика устарела, либо источник нерелевантен. В конспекте помечать такие случаи как `⚠️ нет свежих данных, последнее значение — {год}`.

## Ссылки на источники

Блок ссылок на источники — **обязательная часть материала**, а не опция. Все источники, использованные при подготовке, сохраняй в виде сносок в конце текста, в блоке `## Примечания`:

```
Текст с утверждением[^1]. Ещё одно утверждение[^2].

## Примечания

[^1]: Название источника — URL
[^2]: Название источника — URL
```

- Каждый факт, цифра или цитата из внешнего источника **обязаны** иметь сноску `[^N]` — без сноски цифра в текст не попадает
- Источник и URL берутся из конспекта ресёрчера; автор не выдумывает URL
- Нумерация сносок — сквозная по всему тексту
- Формат: `[^N]: Название источника — URL`
- Блок `## Примечания` защищён `final-trust-editor` от удаления и проверяется оркестратором перед сохранением (`source-attribution.md`). Автопроверка — «check_source_footnotes» (внутренняя автопроверка репозитория, в чате недоступна — проверь этот пункт вручную)

## Не смешивать метрики

Две похожие цифры часто измеряют разное. Это самый распространённый источник фактических ошибок: 7 случаев в анализе отчётов 2026-Q1—Q2. Перед тем как вставить число в текст, прочитай определение в первоисточнике и сверь границы метрики.

### Типовые ловушки

| Смешивают | Правильно различать |
|-----------|---------------------|
| Доля сайтов ↔ доля рынка сертификатов | Let's Encrypt: «~54% сайтов в интернете» (W3Techs) ≠ «~63% выданных SSL-сертификатов» (SSLInsights) |
| DAU ↔ MAU | WeChat: 945 млн MAU (Statista 2024) ≠ 450 млн DAU (устаревшая оценка) |
| MTTI ↔ lifecycle инцидента | «Среднее время обнаружения инцидента 292 дня» (IBM CODB) ≠ «Полный цикл атаки занимает 292 дня» (это lifecycle, не MTTI) |
| Доля атак ↔ доля успешных атак | «80% атак через RDP» (Verizon DBIR — попытки) ≠ «38% успешных проникновений через RDP» (Sophos — итог) |
| Доля рынка ↔ доля внедрений | «Helm используют 75% организаций» (CNCF — adoption) ≠ «Helm — 75% рынка package managers» (нет такого рынка) |
| Госзакупки ↔ локальное производство | «60% по 44-ФЗ/223-ФЗ закупают отечественное» ≠ «60% железа в стране — отечественное» |
| Период: полугодие ↔ год | «406 публикаций утечек в 1H 2025» ≠ «767 млн строк за весь 2025» |
| Точечное событие ↔ серия | «Capital One выплатила $190 млн в 2022» (settlement по иску 2019 года) ≠ «$190 млн ущерб от инцидента» (изначально оценивали в $100-150 млн) |
| Adoption ↔ preferred tool | «используют» ≠ «предпочитают как основной» |
| GitHub stars на дату X ↔ актуальные | Цифры обновляются ежедневно — всегда указывай месяц/год замера |
| Доля сегмента ↔ доля от всех | «5-10% выручки от облака в SMB-сегменте» ≠ «5-10% от всей выручки» |

### Правило: подписывай метрику в скобках

Если есть риск перепутать с похожей метрикой — подписывай в скобках, **что именно** измерено:

```
✗ «Helm используют 75% организаций»
✓ «Helm используют 75% организаций (CNCF Annual Survey 2024 —
   adoption среди опрошенных production-команд)»

✗ «MTTI составляет 292 дня»
✓ «Среднее время обнаружения утечки — 204 дня + 73 дня
   на сдерживание = 277 дней общий цикл (IBM CODB 2024)»
```

### Чек-лист для автора при вставке цифры

Перед каждой цифрой в тексте прогоняй пять вопросов:

1. **Что измерено?** (объект: пользователи / транзакции / компании / выручка / атаки)
2. **Как измерено?** (попытки vs успешные, opt-in vs все, adopt vs prefer)
3. **За какой период?** (месяц / квартал / полугодие / год; на какую дату)
4. **Какая выборка?** (мир / РФ / СНГ; enterprise / все; опрошенные / фактические)
5. **Первоисточник или ретрансляция?** (отчёт CNCF ≠ статья, которая на него ссылается)

Если хотя бы один пункт не проверяем по ссылке — цифра идёт с пометкой `[NEEDS_FACTCHECK: проверить семантику метрики]`.
