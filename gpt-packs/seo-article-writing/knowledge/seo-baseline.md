# Базовые SEO-требования

Минимальные SEO-стандарты для каждой статьи. Детальные конфигурации режимов → навык `seo-knowledge`.

## Ключевые слова

- Плотность основного ключа: 1-2% (не больше)
- Основной ключ обязательно в: H1, первый абзац, 1-2 подзаголовка H2, заключение, meta description
- LSI-слова: 15-25 штук, распределены естественно по тексту
- Не допускай keyword stuffing — читаемость важнее плотности

## Структура для поисковиков

- H1: один на статью, содержит основной ключ
- H2: каждые 300-500 слов, минимум 3 штуки
- H3: для детализации внутри H2-секций
- Списки: минимум 2-3 на статью (numbered или bulleted)
- Таблицы: где данные позволяют структурировать

## Meta-данные

- Title: до 60 символов, основной ключ ближе к началу
- Description: 140-160 символов, ключ + призыв к чтению. Это **сниппет для читателя в выдаче**, а не цель материала: не выноси сюда «обещание читателю» из тезис-карты `editorial-angle` и не описывай статью от третьего лица («Читатель понимает… и подбирает решение»). Шаблон — в секции «Шаблоны заголовков и метаданных» ниже
- H1: отличается от Title, но содержит ключ
- Предложить 2-3 варианта H1 — **только на интерактивном шаге выбора** (`editorial_angle` / сообщение wizard'а) или в companion-отчёте. В тело статьи идёт **один** выбранный H1: блок «## Варианты H1» / «Заголовок (на выбор)» в черновике — блокирующий `editorial-leak` (`ai-pattern-cleanup.md` §8)

## Оптимизация для AI-поиска

- Первые 2-3 предложения каждой секции — самодостаточные (можно процитировать без контекста)
- Определения терминов — в первом упоминании
- Структура «вопрос → ответ» в подзаголовках где уместно
- Факты с числами в отдельных предложениях (не прячь в середине абзаца)

## Внутренние ссылки

- Где уместно — ссылки на документацию VK Cloud
- Формат: `[название сервиса](URL)`, не «нажмите здесь»
- Ссылка на фразе, не на одном слове
- **Если целевой URL ещё неизвестен — не ставь кликабельный якорь** `[текст](#anchor)` на несуществующую секцию: такая заглушка выглядит рабочей ссылкой, но ведёт в никуда. Используй маркер `[NEEDS_LINK: тема]` без markdown-ссылки и продублируй тему в блоке «Почитать по теме». Реальный URL подставляется при публикации

### Блок «Почитать по теме» (перелинковка с корпусом)

В конце статьи — **отдельный блок** `## Почитать по теме` с **2–5 ссылками на уже существующие статьи блога** (не на продукт/доки — это выше, и не вразброс по прозе). Это связывает новый материал с корпусом (тематическая авторитетность, передача веса), не трогая текст.

- **Ссылки не выдумываются.** Кандидаты подбираются движком перелинковки по корпусу — «internal_links» (внутренняя автопроверка репозитория, в чате недоступна — проверь этот пункт вручную) (он возвращает готовый markdown-блок с реальными `/blog/<slug>` и якорями по теме назначения). Оркестратор/автор добавляет блок **после** написания тела.
- **Только релевантное.** Если движок не нашёл подходящих статей (порог ≥2 общих значимых токена) — блок **опускается**, а не наполняется натянутыми ссылками. Лучше нет блока, чем ссылки не по теме. Кандидаты ранжируются весом (свой продукт, pillar по `link_priority`, тип материала, свежесть); статья с `no_related: true` во frontmatter в блок не попадает (opt-out для коммерческих/служебных страниц).
- **Отдельный блок в конце**, не инлайн-ссылки в абзацах: проза не засоряется, перелинковка не выглядит как link scheme.
- В headless/автономном пути тот же движок доступен как функция `internal_links.suggest_for_new(...)` / `related_block(...)`. Контракт корпусного слоя и поток — «content-factory» (внутренний материал системы, в эту выгрузку не входит).
- **Enforcement — блокирующий гейт.** «check_internal_links» (внутренняя автопроверка репозитория, в чате недоступна — проверь этот пункт вручную): сохранённая SEO-статья обязана нести блок «Почитать по теме» **или** ≥ 2 внутренних `/blog/`-ссылки; на изменённых в ветке статьях гейт блокирующий, на legacy-корпусе — advisory. `no_related: true` и короткое тело — легитимный пропуск. Закрывает рецидив «правило есть, но статьи уходят без перелинковки».

## Core Web Vitals (требование к веб-версии, не к тексту)

Это не редакционные правки, а задача для технической/вёрстки-команды — помечай маркером
`⚙️ TECH`, не выдавай за стилистику. Целевые пороги на 75-м перцентиле реальных визитов:

- **LCP** ≤ 2,5 с (загрузка основного контента)
- **CLS** ≤ 0,1 (стабильность вёрстки)
- **INP** < 200 мс (отзывчивость; INP официально сменил FID в марте 2024)

Различай field data (реальные визиты, CrUX — на них опирается Search Console) и lab data
(PageSpeed Insights / Lighthouse). Тяжёлые изображения и встройки — частая причина плохого LCP.

## Шаблоны заголовков и метаданных

Title формируется поисковиком из разных источников, meta description показывается не
гарантированно — но хороший шаблон поднимает шанс качественного сниппета. Подставляй под
инфостиль VK Tech (без кликбейта, ключ ближе к началу):

- **Title:** `[Результат/ответ] для [аудитории] — [контекст/год] | VK Cloud`
- **Title (сравнение):** `[A или B]: что выбрать для [сценария] в [год] | VK Cloud`
- **H1:** `Как [сделать X] в [год]` · `Что такое [термин] и как он работает` · `[A] или [B]: что лучше для [сценария]`
- **Meta description:** `[Короткий ответ]. Разбираем [критерии/шаги], показываем [таблицу/пример/ошибки] и объясняем, кому подходит [вариант].`
- **Intro:** `Коротко: [однозначный ответ в 1–2 предложениях]. Ниже — [что сравним / какие шаги / какие ограничения], чтобы вы могли [целевое действие].`
- **FAQ-пара:** вопрос коротко и разговорно → ответ answer-first в 1–3 предложениях (+ при нужде 1 уточнение, 1 ограничение, 1 ссылка на раздел).

## FAQ-секция: вопросы как реальные поисковые запросы

Блок частых вопросов работает на featured snippets, только если вопросы совпадают с тем, что
люди реально набирают в поиске.

- **Название секции — ключевая фраза, а не «FAQ».** Используй формулировку с темой: «Частые
  вопросы про интеллект-карты», а не «FAQ» или «Частые вопросы».
- **Вопросы = поисковые запросы.** «Как сделать интеллект-карту онлайн бесплатно?» вместо
  маркетингового «Какой сервис выбрать для бизнеса?».
- **3–5 вопросов**, ответ answer-first (формат — в FAQ-паре выше).
