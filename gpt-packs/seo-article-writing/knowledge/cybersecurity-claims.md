# Источники и атрибуция в темах кибербезопасности

Доменное правило для security-тематик. Применяется автоматически, если тема SEO-статьи / пресс-релиза / VK Cloud статьи связана с информационной безопасностью: ИБ, атаки, ransomware, утечки, уязвимости, compliance, PCI DSS / 152-ФЗ / 187-ФЗ / ГОСТ Р 57580.

## Зачем правило

В анализе 21 отчёта генерации обнаружено: **5 из 6 явных галлюцинаций — в темах кибербеза**:
- «90% взломов через SSH» (нет такого факта в первоисточниках)
- «30 000 атак в день» (придуманная цифра)
- «PT 19% XSS 0-day» (вырвана из контекста)
- «Shodan 60% brute-force» (агрегатор без проверки)
- «Pervasive Solutions DRaaS» (компании не существует в указанной форме)
- Индонезия-выкуп — выдуманный кейс

Причина: в security-нише много open-source-блогов и агрегаторов, которые цитируют друг друга по кругу. Автор берёт цифру как факт, фактчекер не успевает развернуть цепочку до первоисточника.

## Запрещённые источники

**Без перепроверки через первоисточник** запрещены:

- `securitymedia.org`, `awisee.com`, `techinsider.ru`, `cnews.ru` (для security-статистики — только как ретрансляция, оригинал обязателен)
- Анонимные Telegram-каналы (даже профильные)
- LinkedIn-посты без ссылки на отчёт
- «Эксперты говорят» / «по мнению специалистов» — без ФИО и компании
- Маркетинговые whitepapers вендоров кибербез-решений — допустимы только методологические части, не цифры

Цифра из такого источника **не идёт в текст**. Допустимо:
1. Найти первоисточник по ссылке внутри статьи
2. Если не найден — пометить `⚠️ только агрегатор, не использовать` в конспекте
3. Если цифра критична для текста, но первоисточника нет — переформулировать как нарратив без числа

## Чек-лист ресёрчера для security-тем

Перед добавлением security-цифры в конспект:

1. **Источник в белом списке?** Если нет — найди эквивалент в белом списке.
2. **Цифра указана с датой публикации источника?** Если нет — отметь `⚠️ нет даты`.
3. **Прямая ссылка на отчёт, а не на статью-обзор?** Цепочку «статья → агрегатор → первоисточник» обязательно развернуть.
4. **Перепроверка на свежесть.** Для CVE-цифр, ransomware-статистики, известных групп — обязательная WebSearch-проверка «{тема} {текущий год}» (например, «BlackCat ransomware 2026» → проверить, не отключена ли группа: BlackCat остановлена в начале 2024).
5. **Семантика цифры понятна?** Что именно измерено — попытки атак, успешные атаки, доля жертв, средний выкуп, время восстановления? (см. «content-quality» (внутренний материал системы, в эту выгрузку не входит) секция «Не смешивать метрики»)

## Чек-лист фактчекера для security-тем

Дополнительно к стандартному фактчекингу:

1. **Группы ransomware:** проверь статус группы на дату публикации. Кейсы из отчётов: BlackCat (отключена февраль 2024), Conti (распалась 2022), REvil (арестована 2022), LockBit (разгромлена февраль 2024). Если статья пишется в 2026, а группа упомянута как «активна» — ⚠️.
2. **CVE:** проверь CVSS-оценку, дату обнаружения и патч-статус на дату публикации. Не используй устаревшие CVE без явного контекста («исторический пример»).
3. **Регуляторика:** ФЗ и приказы отменяются и заменяются. Пример: приказ ФСБ №196 отменён, актуальны 187-ФЗ и новые приказы. Сверяйся с `consultant.ru` / `pravo.gov.ru`.
4. **Инциденты:** даты и пострадавшие компании путают. Пример: «75% систем восстановлено за 3 дня» — Rutube 2022, а не СДЭК 2024.
5. **Инструменты атак:** Shodan-статистика часто завышена в блогах. Проверяй по официальным отчётам Shodan, а не по статьям-обзорам.

## Регуляторные шкалы и классификации — не интерпретировать сверх фактуры

Уровни доверия ФСТЭК (УД1–УД6), классы защищённости ГИС (К1–К3), категории значимости
КИИ (1–3), уровни защищённости ИСПДн (УЗ1–УЗ4) — это **нормативные шкалы**, а не повод
для додумывания. Типовая ошибка автора: продукт сертифицирован по одному уровню, а в
тексте ему приписывают соседние или называют их «ключевыми».

Кейс из генерации (статья «Хранилище для госсектора», 2026-06-09): продукт сертифицирован
по **УД4**, но writer вынес «УД4 и УД2» как «ключевые уровни» и в той же статье указал, что
УД1–3 — для гостайны. УД2 (гостайна) — узкий отдельный контур, на который продукт не
сертифицирован; называть его «ключевым» для хранилища госсектора — фактическая ошибка.

Правила:

1. **Уровень = ровно тот, что в сертификате.** Не приписывай продукту соседние уровни
   шкалы. Если сертификат на УД4 — пиши УД4, не «УД4 и УД2».
2. **Гостайна — отдельный контур.** УД1–3 по Приказу ФСТЭК № 76 относятся только к
   системам с государственной тайной. Не смешивай их с ГИС / ИСПДн / КИИ без гостайны
   (для которых потолок — УД4).
3. **Направление шкалы проверяй.** Чем критичнее система — тем **меньше** номер уровня
   доверия (УД1 — высший). Не путай «выше по защите» и «больше по номеру».
4. **Класс/категорию конкретной системы заказчика не выдумывай.** Если в фактуре её нет —
   пиши, что уровень определяется по результатам категорирования, без конкретной цифры.
5. Любой уровень/класс/категория, которого нет в продуктовом скилле или конспекте, —
   `[NEEDS_FACTCHECK: регуляторная классификация за пределами фактуры]`.

## Что писать, если первоисточник не найден

Не выдумывай. Допустимо:

```
✓ «По данным Positive Technologies за 2025 год, доля атак через RDP
   составляет около 38%[^1]. В то же время Verizon DBIR показывает,
   что попытки подключений по RDP составляют 80% сетевых атак[^2] —
   разница объясняется тем, что Verizon считает попытки, а PT — успешные
   проникновения.»

✗ «По данным экспертов, 90% взломов происходят через SSH»
✗ «Shodan показывает 60% серверов уязвимы к brute-force»
```

Если для важного утверждения нет первоисточника:
- Переформулируй как описание (без процента)
- Запроси у пользователя данные
- Пометь `[NEEDS_FACTCHECK: первоисточник не найден]`

## Подключение правила

Это правило **автоматически** загружается для оркестраторов, если в теме SEO-статьи / пресс-релиза / VK Cloud статьи встречаются триггерные слова:

- ИБ, информационная безопасность, кибербезопасность
- атака, уязвимость, эксплойт, CVE
- ransomware, шифровальщик, выкуп
- утечка, инцидент, breach
- compliance, 152-ФЗ, 187-ФЗ, ГОСТ Р 57580, PCI DSS
- ФСТЭК, уровень доверия, УД1–УД6, реестр Минцифры, КИИ, ГИС, ИСПДн, аттестация, категорирование, air-gap, импортозамещение (для ИТ-систем)
- SOC, SIEM, EDR, NDR, XDR
- DDoS, brute-force, фишинг
- защита, защищённость, защищать (в контексте ИТ-систем)
- penetration testing, pentest, OSINT
- SQL injection, XSS, CSRF, OWASP

Триггеры покрывают темы из отчётов 2026: SQL-injections, S3-security, RDP-security, восстановление инфраструктуры, защита от ransomware.
