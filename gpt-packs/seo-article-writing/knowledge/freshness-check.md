# Свежесть данных и автоматическая re-verification

Правило обязательной проверки свежести цифр. Действует для **ресёрчера**, **автора** и **фактчекера** во всех модулях, **кроме перевода**: у `article-translation-agent` контракт обратный — числа/даты/факты переносятся из оригинала без расхождений, верификатор перевода WebSearch по умолчанию не использует и цифры не «освежает» («translation-standards» (внутренний материал системы, в эту выгрузку не входит), «subagent-protocol» (внутренний материал системы, в эту выгрузку не входит)); устаревшая цифра оригинала — вопрос выбора статьи для перевода, не правок в тексте.

## Зачем правило

В анализе 21 отчёта генерации обнаружено **6 случаев устаревших цифр**, прошедших фактчекинг:
- WeChat: 450 млн DAU (2020) — на момент публикации 945 млн MAU (Statista 2024)
- BlackCat ransomware упомянут как активная группа — фактически отключена февраль 2024
- TAdviser 165 млрд ₽ (рынок S3 в РФ) — устаревшая оценка 2024 vs актуальная 2025
- Приказ ФСБ №196 — отменён, актуальны 187-ФЗ и новые приказы
- TeamViewer-инцидент: упомянут 2024, реально — 2022 (другой инцидент в 2024)
- Higgsfield: дата раунда инвестиций перепутана между сентябрём 2025 и январём 2026

Существующее правило `research-knowledge-base.md` ограничивает только переиспользование исследований (< 30 дней). Этого недостаточно: цифра может быть «свежей» в смысле момента сбора, но фактически устаревшей в смысле первоисточника.

## Правило свежести по типам данных

| Тип данных | Допустимая давность | Действие при превышении |
|-----------|---------------------|--------------------------|
| Рыночная статистика (объём рынка, доли, темпы роста, CAGR) | 12 месяцев | re-verification: WebSearch «{тема} {текущий год}» |
| MAU / DAU продуктов | 12 месяцев | re-verification |
| GitHub stars / contributors / downloads | 3 месяца | re-verification (обновляются ежедневно) |
| Adoption технологии (CNCF Survey и аналоги) | 12 месяцев | re-verification |
| Версии ПО, EOL-даты | На момент публикации | проверка vendor changelog |
| **Редакция версионируемого стандарта** (OWASP Top 10, Kubernetes Gateway API, CIS, PCI DSS…) | На момент публикации | проверка наличия новой редакции — см. ниже |
| CVE и патч-статус | На момент публикации | проверка NVD / vendor advisory |
| Регуляторика РФ (ФЗ, приказы, ГОСТ) | На момент публикации | проверка consultant.ru / pravo.gov.ru |
| Регуляторика РК | На момент публикации | проверка adilet.zan.kz |
| Группы ransomware / APT / статус | 6 месяцев | re-verification (группы отключают, переименовывают) |
| Раунды инвестиций, M&A | На момент события | разделять этапы (раунд A → раунд B) |
| Исторические факты (год выхода Kubernetes 1.0 и т.п.) | Без ограничения | проверка не требуется |

### Версионируемые стандарты: primary = текущая редакция

Для стандартов и спецификаций, выходящих редакциями (OWASP Top 10, Gateway API,
CIS Benchmarks и т.п.), материал строится **на текущей редакции как основной**.

- **Перед написанием** ресёрчер/фактчекер проверяет, не вышла ли свежая редакция
  (для OWASP, K8s — блог проекта; для регуляторики — официальный источник).
- **«Старую ещё требуют в комплаенсе» — повод упомянуть, а не строить на ней
  весь материал.** Legacy-редакция идёт отдельным блоком «что изменилось», не
  каркасом статьи.
- Кейс: статья про OWASP Top 10 была написана на редакции **2021**, хотя на
  момент генерации действовала **2025** (фактчекер знал про релиз, но принял
  рационализацию «2021 базовая для compliance»). Публикация переписала под 2025
  целиком. Рационализация «старая ещё в ходу» не оправдывает устаревший каркас.

## Контракт ресёрчера

### Маркировка свежести

В конспекте каждая цифра обязана иметь поле «дата первоисточника»:

```
✅ 75% организаций используют Helm — CNCF Annual Survey 2024,
   опубликовано декабрь 2024, актуально на дату ресёрча {YYYY-MM-DD}

⚠️ нет свежих данных: последняя цифра WeChat DAU за 2020 год,
   за 2024-2025 нашёл только MAU 945 млн (Statista). Рекомендация:
   использовать MAU вместо DAU, либо переформулировать без цифры
```

### Запрос текущего года

При WebSearch ресёрчер **обязан** формулировать запросы с явным указанием года:

```
✅ «облачный рынок Россия 2026»
✅ «Helm adoption 2026 site:cncf.io»
✅ «PostgreSQL 16 release notes 2025»

✗ «облачный рынок Россия»  ← без года → может попасть статья 2022
✗ «Helm adoption»          ← без года → старая CNCF survey
```

Если запрос за текущий год не дал результатов — переход на прошлый год **с явным флагом**: «по данным за {год-1}, более свежих данных нет».

### Красный флаг «только старые данные»

Если для конкретной метрики ресёрчер находит только данные старше 2 лет — это **красный флаг**:

1. Возможно, метрика устарела (продукт сменил KPI, методология поменялась)
2. Возможно, источник перестал публиковать данные (отчёт прекращён)
3. Возможно, нужна другая формулировка вопроса

В этом случае ресёрчер **обязан** в конспекте указать:

```
⚠️ Данные за последние 2 года не найдены. Последняя точка: {год} ({источник}).
Рекомендация автору: переформулировать без числа / запросить у пользователя /
использовать как исторический контекст с явной датой
```

## Контракт автора

### Обязательная пометка года для старых цифр

Любая цифра старше **12 месяцев** требует явной пометки года/даты в тексте. Без пометки — нельзя.

```
✗ «WeChat имеет 450 млн пользователей»                  ← устаревшее
✓ «WeChat имел 450 млн DAU по оценке 2020 года; на 2024
   платформа отчиталась о 945 млн MAU[^1]»

✗ «Российский рынок S3-хранилищ — 165 млрд ₽»            ← без даты
✓ «По оценке TAdviser за 2024 год, российский рынок объектных
   хранилищ составил 165 млрд ₽[^2]; данные за 2025 пока
   не опубликованы»
```

### Запрет на старые цифры без контекста

Цифру старше **2 лет** автор **не имеет права** использовать как актуальное значение. Допустимо только в паре «было → стало» с явными датами:

```
✓ «С 2022 года рынок вырос с X (2022, TAdviser) до Y
   (2024, IDC) — рост на N%»

✗ «Рынок составляет X» (где X — оценка 2022 года)
```

## Контракт фактчекера

### Чек-лист актуализации перед содержательной проверкой

Перед проверкой содержательных утверждений фактчекер **обязан** пройти чек-лист актуализации:

1. **Каждая цифра имеет дату источника в тексте?** Если нет — корректировка: добавить.
2. **Дата источника не старше лимита для этого типа данных?** (см. таблицу выше) Если старше — WebSearch обновить.
3. **Версия ПО / релиз / EOL — на момент публикации актуальны?** Сверить vendor changelog.
4. **Регуляторные ссылки активны?** Сверить consultant.ru / adilet.zan.kz.
5. **Группы ransomware упомянуты как активные?** Сверить с последними отчётами (Mandiant, CISA, PT).
6. **Раунды/события — правильная дата и сумма?** Не путать этапы.

### Обязательный повторный WebSearch для цифр старше 12 месяцев

Возраст цифры фактчекер определяет **по тексту** — он работает в изоляции и конспекта не видит (`stage-isolation.md`): год в формулировке («по данным за 2024»), датировка источника в сноске `[^N]`, датируемые сущности (версия ПО, событие). Цифра без определимого возраста — кандидат на `[NUMBER_RECHECK]`. Для каждой цифры старше 12 месяцев фактчекер **обязательно** делает WebSearch с запросом «{тема} {текущий год}»:

- Если нашёл свежую цифру — корректировка: замена в тексте.
- Если не нашёл — оставить как есть, но обязательно с пометкой года в тексте.
- Если данные за текущий год противоречат старым — отметить в отчёте как «⚠️ Динамика: {старая} → {новая}».
- **Если поискового инструмента НЕТ** (рантайм без web-tool; модель с нативным
  поиском, которой function-tools отключены; офлайн-прогон) — обязанность не
  исчезает, а деградирует честно: пометить цифру `[NUMBER_RECHECK: свежесть не
  сверена — нет поискового инструмента]` и перечислить такие места в отчёте.
  Молча пропустить сверку нельзя; выдумать «свежую» цифру — тем более.

### Запрет на «округление» при актуализации

Если фактчекер заменяет старую цифру на новую — **не округлять**. Если старая «28K stars», новая «29.6K» — записывать как «29,6K stars» (точная цифра из источника), не «~30K».

## Защита в pipeline

Финальная траст-редактура (`final-trust-editor`) защищает **точное значение цифры** от изменения. Это означает: если ресёрчер и автор сделали свою работу правильно (свежая цифра + пометка года), редактор не имеет права её округлять или схлопывать в диапазон. См. `final-trust-editing.md`.

## Слой автопроверки

«check_freshness» (внутренняя автопроверка репозитория, в чате недоступна — проверь этот пункт вручную) (внутренний гейт репозитория (в чате недоступен), advisory REVIEW) — детектор
риска: находит числовые метрики и атрибуции данных, опирающиеся на год старше
2 лет без свежего значения рядом. Он НЕ заменяет чек-лист фактчекера выше
(детектор не знает, устарела ли цифра фактически) — он гарантирует, что такие
места не пройдут незамеченными. FP легитимны (исторические сравнения) —
поэтому advisory, не блок.

## Связь с другими правилами

**Это правило — ПЕРВОИСТОЧНИК порогов свежести.** Раздел «Приоритет
актуальности данных» в `content-quality.md` — рабочая выжимка для
автора/ресёрчера; пороги правятся сначала здесь, выжимка приводится следом
(иначе копии разъезжаются молча — уже случалось с порогом 6/12 месяцев).

- `source-attribution.md` — атрибуция (год — обязательная часть атрибуции)
- «content-quality» (внутренний материал системы, в эту выгрузку не входит) — приоритет актуальности данных (выжимка этого правила), секция «Не смешивать метрики»
- `final-trust-editing.md` — защита цифр от изменения
- «research-knowledge-base» (внутренний материал системы, в эту выгрузку не входит) — свежесть переиспользуемых исследований
- `source-standards.md` — иерархия Tier 1-7 (свежесть как часть оценки источника)
- `cybersecurity-claims.md` — особые требования к свежести для security-данных
